# 위챗 공개계정 글: 작가-편집자-독자 3 에이전트 협업

$ARGUMENTS 에 대해 심층 리서치를 수행하고, 바로 발행할 수 있는 위챗 공개계정 글을 산출한다. 세 에이전트가 각자의 역할을 담당한다: 작가가 깊이 있는 초고를 쓰고, 편집자가 구조와 표현을 다듬으며, 독자가 목표 독자층 관점에서 검토한다.

**지원 입력 형식**: 주제 설명, 예: `대형 모델 OPD 기술 해설`、`Qwen3 기술 보고서 해설`、`왜 버핏은 기술주를 사지 않는가`

---

## 설계 원칙

좋은 공개계정 글은 세 가지 차원을 동시에 충족해야 한다:
1. **깊이**——끝까지 읽는 사람에게 보답한다 (작가 담당)
2. **가독성**——구조가 명확하고, 흐름이 좋으며, 포기하지 않게 한다 (편집자 담당)
3. **실제로 이해 가능**——목표 독자가 중간에 포기하지 않는다 (독자 담당)

혼자 쓰면 '자기만족'에 빠지기 쉽다——쓴 사람은 명확하다고 느끼지만, 읽는 사람은 이해하지 못한다. 3 에이전트 협업의 본질은 **외부 관점을 강제로 도입**하는 것이다.

---

## 1단계: 리서치 및 소재 수집

### 1번째 스텝: 글의 포지셔닝 확정

작성을 시작하기 전에 아래 정보를 확인한다 (사용자가 지정하지 않으면 직접 질문한다):

| 차원 | 확인 필요 항목 | 기본값 |
|------|---------|--------|
| **목표 독자** | 기술 배경 수준 | 약간의 기술 배경은 있지만 해당 분야 전문가는 아님 |
| **글의 깊이** | 교양/중간 깊이/심층 | 중간 깊이 (공식은 있지만 설명을 충분히 함) |
| **글의 길이** | 글자 수 범위 | 3000-4000자 |
| **원본 논문/자료 다운로드 필요 여부** | PDF/이미지 필요 | 예 |
| **작성 스타일** | 격식체/대화체/직설적 | 대화체 (똑똑한 친구에게 쓰는 것처럼) |

### 2번째 스텝: 심층 리서치

Agent 도구를 사용해 리서치 에이전트 2-3개를 **병렬**로 실행하여 충분한 소재를 수집한다:

**리서치 에이전트 A: 핵심 내용 리서치**
- 논문 해설이면: 논문 PDF 다운로드, 핵심 기여·주요 도표·실험 결과 추출
- 기술 주제이면: 최신 동향·핵심 논문·기술 세부사항 검색
- 비즈니스/투자 주제이면: 최신 데이터·업계 리포트·경쟁 구도 검색

**리서치 에이전트 B: 업계 배경 및 적용 사례**
- 해당 기술/주제의 업계 적용 현황 검색
- 어느 기업이 쓰고 있는가? 효과는 어떤가?
- 최신 발전 추세와 이정표 사건

**리서치 에이전트 C (선택): 경쟁제품/비교 리서치**
- 유사 방법/제품 비교
- 역사적 발전 맥락
- 미래 발전 방향

### 3번째 스텝: 소재 프레임워크 정리

모든 리서치 에이전트가 완료되면 다음을 정리한다:
1. **핵심 논점** (글이 전달하려는 핵심 메시지를 한 문장으로 요약)
2. **핵심 데이터** (임팩트 있는 데이터 포인트 3-5개)
3. **이미지 목록** (어떤 이미지가 필요한지, 출처는 무엇인지)
4. **글 목차** (6-8개 섹션의 제목과 핵심 내용)

## 2단계: 작가 에이전트가 초고 작성

Agent 도구를 사용해 **작가 에이전트**를 실행하고, 상세한 작성 지시를 제공한다.

### 작가 에이전트 프롬프트 템플릿

```
당신은 심층 기술 글쓰기 전문가입니다（Author Agent）. 위챗 공개계정 글을 작성해야 합니다.

## 목표 독자
{1단계에서 확인한 독자 페르소나에 따라}

## 작성 스타일 요구사항
- 순수 한국어로 표현, 한국어·영어 혼용 지양 — 기술 용어는 첫 등장 시 영문 병기 후 한국어 사용
- 학술 논문 번역이 아닌, 똑똑한 친구에게 쓰는 기술 교양글
- 비유로 이해를 돕되, 비유는 적절하고 진부하지 않게
- 핵심 공식/데이터는 포함하되, 모든 항목을 평이한 언어로 설명
- 이모지 사용 금지
- 단락은 4줄 초과 금지 (공개계정 열람 환경)

## 핵심 내용
{정리된 소재, 데이터, 논거}

## 글 구조 요구사항
1. **도입부 (앞 3단락)**: 강력한 훅 필수 — 데이터 충격 또는 반직관적 결론으로 시작, 온화한 비유로 시작하지 않음
2. **배경**: 왜 이것이 중요한가? 어떤 문제를 해결하는가?
3. **핵심 내용 (2-3절)**: 기술적 깊이를 여기서 드러내되, 모든 기술 포인트에 "평이한 설명" 첨부
4. **실증/사례**: 데이터와 사례로 말하고, 공허한 이론 금지
5. **업계 영향/전망**: 이것이 업계에 갖는 의미
6. **결말**: 전파력 있는 한 문장 판단으로 마무리, 캡처 후 공유 적합

## 이미지 요구사항
- 논문 해설류 글: 논문 PDF에서 원본 이미지를 반드시 추출하여 `![설명](상대경로)` 형식으로 직접 삽입, [그림X: 설명] 자리표시자 금지
- 추출 방법: pdftoppm으로 PDF 페이지를 고해상도 PNG로 렌더링(최소 900 DPI), PIL로 목표 도표 영역 크롭
- 이미지 1장당 최소 500KB, 고화질 보장
- 이미지는 `assets/{주제약어}/` 디렉터리에 통일 저장
- 비논문류 글: 이미지가 필요하면 적합한 이미지를 검색·다운로드하여 동일하게 직접 삽입

## 공식 요구사항
- 모든 수식은 LaTeX 형식: 인라인은 `$...$`, 독립 수식은 `$$...$$`
- 순수 텍스트 수식 작성 금지 (예: `> D_KL(P || Q) = ...`), 반드시 LaTeX 렌더링 형식 사용
- 모든 수식 뒤에 "쉽게 말하면……" 번역 단락 추가

완전한 글 초고를 작성하십시오, 약 {목표 글자 수} 자.
```

### 작가 에이전트 완료 후

초고 파일이 생성되었는지 확인하고, 전문을 읽어 내용의 완결성을 확인한다.

## 3단계: 편집자 에이전트 + 독자 에이전트 병렬 검토

초고가 완성되면, Agent 도구를 사용해 **같은 메시지에서** 편집자 에이전트와 독자 에이전트를 실행한다.

### 편집자 에이전트 프롬프트 템플릿

```
당신은 경험 많은 공개계정 편집자입니다（Editor Agent）. 아래 글에 대해 정밀 검토를 수행하십시오.

## 검토 기준
1. **제목**: 카카오톡/SNS에서 클릭을 유도하는가? 너무 길어 잘리지 않는가 (30자 초과)?
2. **도입부**: 앞 3단락이 독자를 붙잡는가? 훅이 충분히 강한가?
3. **구조**: 논리 흐름이 매끄러운가? 건너뜀이나 단절이 있는가?
4. **깊이와 가독성 균형**: 공식/기술 부분이 진짜 쉽게 설명되어 있는가? "쉬운 척하지만 실제로 설명이 안 된" 부분은 없는가?
5. **리듬**: 너무 긴 단락은 없는가? 각 절의 길이가 적절한가?
6. **이미지**: 이미지가 실제로 삽입되어 있는가 (자리표시자가 아닌)? 독자에게 시각 보조가 가장 필요한 지점에 위치하는가?
7. **결말**: 전파력이 있는가? 독자가 읽고 나서 공유하고 싶어지는가?

## 글 전문
{완전한 초고}

## 출력 형식
1. 전체 평가 (3-5문장)
2. 제목 수정 제안 (2-3개 대안 제시)
3. 절별 수정 제안 ("원문 → 수정 제안" 대조 형식으로 구체적으로)
4. 가장 중요한 개선점 3가지
```

### 독자 에이전트 프롬프트 템플릿

```
당신은 {목표 독자 페르소나}입니다（Reader Agent）. 독자 관점에서 아래 글을 검토하십시오.

## 당신의 배경
{목표 독자의 지식 수준과 읽기 습관을 구체적으로 설명}

## 글 전문
{완전한 초고}

## 아래 질문에 답하십시오
1. 앞 3단락을 읽고 나서 계속 읽고 싶었는가? 왜?
2. "이해가 안 되거나" "다시 읽어야 했던" 부분은 어디인가? 어느 문장인가?
3. 기술/공식 부분을 이해했는가? "평이한 설명"이 도움이 되었는가?
4. 글의 핵심 비유가 적절한가? 더 나은 비유가 있는가?
5. 너무 긴가, 너무 짧은가? 어느 지점에서 집중력이 떨어지는가?
6. 다 읽고 나서 한 문장으로 글의 핵심 주장을 요약할 수 있는가?
7. 이 글을 공유하겠는가? 공유할 때 어떤 말을 붙이겠는가?
8. 알고 싶었지만 글에서 다루지 않은 질문이 있는가?
```

## 4단계: 최종본 확정

### 1번째 스텝: 두 에이전트의 피드백 종합

다음 고빈도 문제에 중점을 둔다:

| 문제 유형 | 편집자 공통 피드백 | 독자 공통 피드백 | 처리 방식 |
|---------|------------|------------|---------|
| 도입부 약함 | 훅이 약함 | 앞 3단락에서 계속 읽을 동기 없음 | 데이터/반직관적 결론으로 도입부 재작성 |
| 기술 섹션에서 포기 | 공식이 너무 빽빽함 | 특정 단락을 3번 다시 읽어야 함 | 공식 삭제 또는 이미지화, 더 직관적인 비유 추가 |
| 흐름이 느림 | 특정 섹션이 너무 길음 | 특정 부분에서 집중력 상실 | 합치거나 삭제 (특히 후반부 반복되는 기술 설명) |
| 결말이 약함 | 전파력 부족 | 공유하고 싶지 않음 | 캡처해서 공유할 수 있는 한 문장 판단으로 재작성 |
| 개념이 건너뜀 | 논리에 단절 있음 | 특정 부분에서 '갑자기 이해 안 됨' | 연결 문장 또는 배경 설명 추가 |

### 2번째 스텝: 수정 실행

피드백에 따라 글을 재작성한다. 핵심 수정 원칙:

1. **편집자와 독자 모두 지적한 문제는 반드시 수정**
2. **편집자만 지적한 문제는 대부분 수정** (편집자의 전문적 판단은 대체로 정확함)
3. **독자만 지적한 문제는 상황에 따라 수정** (독자 피드백은 실제 경험을 반영하지만, 모든 항목에 응답할 필요는 없음)
4. **두 의견이 충돌할 때는 독자 우선** (편집자는 완벽을 추구하지만, 독자 경험이 최종 기준)

### 3번째 스텝: 이미지 추출

논문 해설류 글은 최종본 확정 전에 이미지 추출을 완료해야 한다:

1. **렌더링**: `pdftoppm -png -r 900 -f {페이지} -l {페이지} 논문.pdf /tmp/page` (900 DPI부터 시작, 이미지가 500KB 미만이면 1200 또는 1500 DPI로 올림)
2. **위치 확인**: 150 DPI로 전체 페이지 렌더링 후 육안으로 각 도표의 픽셀 좌표 확인
3. **크롭**: PIL로 좌표에 따라 크롭, `compress_level=1`로 저장, 각 이미지 ≥ 500KB 확인
4. **저장**: `assets/{주제약어}/` 디렉터리에 저장, `fig{번호}-{설명}.png`로 명명
5. **삽입**: 글에서 `![설명](../../assets/{주제약어}/fig{번호}-{설명}.png)`로 참조

### 4번째 스텝: 최종 파일 산출

최종본을 md 파일로 저장하고, 파일 끝에 논문/자료 원문 링크를 첨부한다:

```markdown
**논문 원문:**
- arXiv: {링크}
```

## 작성 금지선

1. **데이터 날조 금지**. 인용 데이터는 반드시 출처가 있어야 하고, 찾지 못하면 "추정"으로 표기
2. **AI 어투 금지**. "함께 살펴보겠습니다", "주목할 만한 것은", "말하지 않을 수 없는" 등의 상투어 금지
3. **과도한 약속 금지**. 기술 글에서 "혁명적인", "파괴적인" 표현 금지, 데이터로 말함
4. **공식에는 반드시 평이한 설명 첨부**. 모든 공식 뒤에는 "쉽게 말하면……"이라는 번역 단락 필요
5. **공식은 반드시 LaTeX 사용**. `$$...$$` 형식, 순수 텍스트 공식 금지
6. **이미지는 반드시 실제 삽입**. 논문 해설류는 PDF에서 고해상도 원본 이미지 추출 (≥500KB), `[그림X]` 자리표시자 금지
7. **표 괄호 주석은 정확하게**. 개념을 설명할 때 정확한 정의 사용, 모호한 동빈구 금지
8. **비유는 일관성 유지**. 전문에서 하나의 주선 비유를 사용하고, 섹션마다 새 비유로 바꾸지 않음
9. **결말은 반드시 전파력 있게**. 마지막 문장은 단독으로 캡처해서 공유할 가치가 있어야 함
