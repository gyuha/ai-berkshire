# 투자 리서치 팀: 4역할 병렬 분석 프레임워크

$ARGUMENTS에 대해 팀 기반 투자 리서치 분석을 수행한다. Team 도구를 사용해 실제 멀티 에이전트 병렬 리서치 팀을 구성한다.

## 실행 흐름

### 1단계: 팀 프레임워크 제시

다음 팀 구성을 사용자에게 보여주고 확인 후 시작한다:

| 역할 | 담당 | 분석 프레임워크 |
|------|------|----------------|
| **team-lead**(본인) | 총괄 조율, 분석 통합, 최종 보고서 출력 | 4대 투자 대가 종합 프레임워크 |
| **business-analyst** | 비즈니스 모델 & 경제적 해자 분석 | 돤융핑 시각 |
| **financial-analyst** | 재무제표 & 가치평가 분석 | 워런 버핏 시각 |
| **industry-researcher** | 산업 구조 & 경쟁 동향 | 찰리 멍거 시각 |
| **risk-assessor** | 리스크 평가 & 경영진 판단 | 리루 시각 |

### 1.5단계: AI 리서치 편향 평가

팀 생성 전, 해당 기업의 "AI 리서치 가능성" 평가를 사용자에게 먼저 제시한다:

**정보 풍부도 등급** (리서치 전략 결정):
| 등급 | 특징 | 리서치 전략 조정 |
|------|------|----------------|
| A등급 (정보 충분) | 상장 다년, 광범위한 애널리스트 커버리지 | 팀이 **반론 검증**과 **비컨센서스 시각**에 집중 — 시장 컨센서스와 일치하는 "맞는 말이지만 의미 없는" 분석 회피 |
| B등급 (정보 보통) | 상장 기간 짧음, 커버리지 제한적 | 각 에이전트의 추산 데이터에 신뢰도 표기 필수, team-lead 통합 시 "데이터 충분도" 표기 |
| C등급 (정보 희소) | 비주류/신규 상장/신흥 시장 | 팀이 "제1원칙 모드"로 전환: 보고서 완성도를 추구하지 않고 비즈니스 본질에 관한 핵심 질문 몇 가지에 집중 |

**핵심 주의사항**: 자료가 많다고 확실성이 높은 게 아니고, 자료가 적다고 확실성이 낮은 게 아니다. AI가 출력할 수 있는 신뢰도 ≠ 투자의 실제 확실성. 확실성은 비즈니스 모델 자체에서 나오지, 자료의 양에서 나오지 않는다.

각 에이전트에게 등급 결과를 전달해 리서치 방식에 반영시킨다.

### 2단계: 팀 생성

TeamCreate로 팀을 생성한다:
- team_name: `{회사명}-research` (영문 소문자, 예: `meituan-research`)
- agent_type: `team-lead`

### 3단계: 태스크 4개 생성

TaskCreate로 다음 4개 태스크를 생성한다 (각 태스크에 subject, description, activeForm 포함):

#### 태스크 1: 비즈니스 모델 분석
- subject: `{회사명} 비즈니스 모델·경제적 해자·고객 가치 분석`
- description 포함 내용:
  1. 비즈니스 모델 본질: 핵심 사업 정의, 수익 구조 분해
  2. 플랫폼/제품 플라이휠 효과 작동 방식
  3. 경제적 해자 분석: 브랜드/전환 비용/네트워크 효과/규모의 경제/기술 장벽 — 항목별 검증
  4. 고객 가치: 각 이해관계자에게 창출하는 고유 가치
  5. 사업 포트폴리오와 시너지 효과
  6. 돤융핑의 "좋은 사업" 기준 평가: 차별화, 가격결정력, 지속 가능한 경쟁우위
  7. 최신 실적 보고서·산업 보고서 등 공개 정보 검색 필수

#### 태스크 2: 재무 및 가치평가 분석
- subject: `{회사명} 재무 데이터·수익성·가치평가 분석`
- description 포함 내용:
  1. 최근 3~5년 매출, 순이익, 영업이익 추이
  2. 수익성 지표: ROE, ROA, 매출총이익률, 영업이익률
  3. 현금흐름 분석: 영업현금흐름, 잉여현금흐름(FCF), 자본적 지출
  4. 재무상태표 건전성: 현금 보유, 부채비율, 유동성
  5. 가치평가 분석: PE/PS/PB/EV 등, 과거 및 동종업계 비교
  6. 안전마진 평가: 내재가치 vs 현재 주가
  7. **재무 정확성 검증 (반드시 Bash로 도구 호출, 암산 금지)**:
     - 시가총액 검증: `python3 ~/ai-berkshire/tools/financial_rigor.py verify-market-cap --price {가격} --shares {주식수} --reported {보고 시가총액} --currency {통화}`
     - 가치평가 검증: `python3 ~/ai-berkshire/tools/financial_rigor.py verify-valuation --price {가격} --eps {EPS} --bvps {주당순자산}`
     - 핵심 데이터 교차 검증: `python3 ~/ai-berkshire/tools/financial_rigor.py cross-validate --field {필드} --values '{JSON}' --unit {단위}`
     - 3시나리오 가치평가: `python3 ~/ai-berkshire/tools/financial_rigor.py three-scenario --price {가격} --eps {EPS} --shares {주식수억} --growth {낙관} {중립} {비관} --pe {낙관PE} {중립PE} {비관PE}`
     - 도구 출력 결과를 검증 기록으로 보고서에 직접 삽입

#### 태스크 3: 산업 및 경쟁 분석
- subject: `{산업} 산업 구조 및 {회사명} 경쟁 태세 분석`
- description 포함 내용:
  1. 산업 규모와 성장: 시장 규모, 성장률, 침투율
  2. 경쟁 구도: 주요 경쟁사 시장점유율, 경쟁 전략 비교
  3. 핵심 경쟁자 위협 평가: 주요 경쟁사 개별 분석
  4. 세부 세그먼트별 경쟁 구도
  5. 산업 트렌드: 기술 변화, 규제 영향, 신규 진입자
  6. 가치사슬 분석: 상류·중류·하류 가치 배분
  7. 최신 산업 데이터 및 경쟁 동향 검색 필수

#### 태스크 4: 리스크 및 경영진 평가
- subject: `{회사명} 투자 리스크·경영진 질 평가`
- description 포함 내용:
  1. 경영진 평가: CEO 능력범위, 진정성, 전략적 안목, 자본 배분 능력, 과거 의사결정 품질
  2. 규제 리스크: 현재 및 잠재적 규제 영향
  3. 경쟁 리스크: 각 경쟁사 위협 수준 평가
  4. 사업 리스크: 신사업 적자, 확장 불확실성
  5. 거시 리스크: 경기 사이클, 산업 사이클 영향
  6. 지배구조: 지분 구조, 특수관계인 거래, 주주 환원 정책
  7. 장기 확실성: 10년 후 이 기업은 어떻게 될까? 무엇이 비즈니스 모델을 파괴할 수 있는가?
  8. 최신 규제 동향, 경영진 발언 등 검색 필수

### 4단계: 에이전트 4개 병렬 실행

Task 도구로 에이전트 4개를 동시에 실행한다 (**반드시 같은 메시지에서 병렬 호출**):

각 에이전트 설정:
- `subagent_type`: `general-purpose`
- `run_in_background`: `true`
- `team_name`: 해당 팀명
- `name`: 해당 역할명 (business-analyst / financial-analyst / industry-researcher / risk-assessor)

각 에이전트 프롬프트 템플릿:

```
당신은 {회사명} 투자 리서치 팀의 "{역할 한국어명}"으로, {대가 이름} 투자 시각에서 {회사명}을 분석한다.

태스크 #{태스크 번호}를 완료하라: {태스크 subject}

세부 요건:
{태스크 description 내용}

**리서치 방법**:
- WebSearch로 최신 공개 정보(실적 보고서, 산업 보고서, 뉴스) 검색
- **재무 데이터는 반드시 두 개의 독립 출처에서 확보**, skills/financial-data.md 규정 준수 (미국주식: macrotrends+stockanalysis; 홍콩주식: aastocks+macrotrends; A주: 동방재부+거조자신), 두 출처 오차 >1%이면 표기
- 데이터 정확성 확보, 핵심 데이터에 출처 표기
- 표면적 분석을 넘어 깊이 있게 분석

**출력 요건**:
- 보고서는 상세하게, 핵심 데이터는 마크다운 표로 제시
- 각 분석 차원마다 명확한 결론과 평점 제시
- 보고서 말미에 해당 차원의 종합 결론 포함

**완료 후**:
1. TaskUpdate로 태스크 #{태스크 번호}를 completed로 표시
2. SendMessage로 완전한 분석 보고서를 team-lead에게 전송 (type: "message", recipient: "team-lead")
```

### 5단계: 보고서 수신 및 진행상황 추적

- 어떤 에이전트가 완료됐고 어떤 에이전트가 아직 리서치 중인지 실시간 진행 현황을 사용자에게 보여준다
- 보고서를 받을 때마다 진행 상황을 업데이트하고 해당 보고서의 핵심 포인트(3~5개)를 제시한다
- 4개 보고서가 모두 도착할 때까지 대기한다

### 6단계: 팀원 종료

보고서를 모두 받은 후, 4개 에이전트에게 shutdown_request를 전송한다 (SendMessage 사용, type: "shutdown_request").

### 7단계: 최종 보고서 통합

4개 분석 보고서를 종합해 다음 구조의 최종 보고서를 출력한다:

---

#### 1. 한 줄 결론
> 한 단락(50~100자)으로 투자 가치 여부와 핵심 논리를 요약

#### 2. 4차원 평점 총표
| 차원 | 프레임워크 | 평점(1~5★) | 핵심 판단 |
|------|-----------|-----------|----------|

종합 평점: X / 5

#### 3. 핵심 데이터 요약
주요 재무·경영 지표 표 (최근 2년 비교)

#### 4. 차원별 분석 요약
각 차원에서 가장 중요한 발견 3~5개 추출

#### 5. 투자 논점 (Bull vs Bear)
- 🟢 강세 논리 (5~7개)
- 🔴 약세 논리 (5~7개)

#### 6. 워런 버핏 매수 전 체크리스트
| # | 점검 항목 | 통과? | 설명 |
10개 핵심 항목 개별 평가

#### 7. 최종 투자 의견
- 정성적 판단표 (사업 품질/경영진/가치평가/타이밍)
- 투자 성향별 운용 의견표 (공격형/안정형/보수형 → 의견+가격 범위)
- 핵심 촉매 (비중 확대 신호/비중 축소 신호 각 3~5개)

#### 8. 요약 단락
최종 요약 100~200자

---

### 8단계: 보고서 저장

완성된 최종 보고서를 `~/{회사명}투자연구보고서_{날짜}.md` (날짜 형식 YYYYMMDD)에 저장한다.

### 9단계: 데이터 표본 검사 (게시 전 검수 절차)

```bash
# Step 1 — 표본 검사 목록 추출 (15% 무작위 샘플링)
python3 ~/ai-berkshire/tools/report_audit.py extract \
  --report <보고서 파일 경로>

# Step 2 — 목록 각 항목을 신뢰할 수 있는 출처에서 데이터 확인 (skills/financial-data.md 참조)

# Step 3 — 통과/반려 판정 출력
python3 ~/ai-berkshire/tools/report_audit.py verdict \
  --results '<작성된 JSON>' \
  --report <보고서 파일명>
```

**[통과]** 전부 통과 → 보고서 발행 가능; **[반려]** 불통과 항목 있음 → 수정 후 재검토.

### 10단계: 팀 정리

TeamDelete로 팀 리소스를 정리한다.

## 중요 주의사항

1. **에이전트 4개는 반드시 병렬 실행** — 같은 메시지에서 Task 도구 4회 호출
2. **에이전트는 SendMessage로 보고** — 파일 협업이 아닌 메시지 통신
3. **데이터 정확성** — 에이전트에게 WebSearch로 최신 데이터 검색 요구, 핵심 데이터 교차 검증
4. **결론은 명확하게** — 매수/관망/회피 의견과 구체적 가격 범위 제시 회피 금지
5. **모든 분석은 데이터로 뒷받침** — 데이터 출처 첨부
6. **인내심을 갖고 대기** — 에이전트 4개 리서치에 수분 소요, 사용자에게 실시간 진행 현황 업데이트
7. **반편향 인식** — team-lead가 통합 시 반드시 평가: 각 에이전트의 분석이 자료 충분도에 의해 제한됐는가? 시장 컨센서스와 과도하게 일치하는가? 최종 보고서에 "정보 풍부도 등급"과 "AI 리서치 한계 선언" 포함 필수
8. **정보 희소 시 정직 원칙** — 보고서에 "데이터 부족"을 공백으로 표기할지언정, 추측으로 프레임워크를 채워 확실성을 위장하지 말 것
